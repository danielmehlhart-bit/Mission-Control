# Mission Control Voice Call Mode V1 Plan

Date: 2026-05-08

## Goal

Build the first real Call Mode V1 iteration on top of the existing Mission Control Voice backend. `/voice` remains the main route, but the primary surface shifts from a diagnostic Voice Console to a mobile-first Hermes call screen.

## Constraints

- Reuse the existing voice session, profile, context, Realtime, tool, turn, event, memory, and Telegram bridge infrastructure.
- Keep the old transcript/text workflow available as fallback/details, not as the primary surface.
- Do not expose raw internal context or secrets through API responses.
- Do not delete legacy implementation paths blindly.
- Do not commit in this goal run.

## Implementation Steps

1. Normalize Call Mode profiles and bindings.
   - Keep V1 button-ready profiles to `main`, `sales_support`, and `luma`.
   - Bind `main` to Telegram DM `485318478`.
   - Bind `sales_support` to Telegram chat `-1003998265477`, topic/message `23`.
   - Bind `luma` to Telegram chat `-1003998265477`, topic/message `24`.

2. Add Call Mode session flags where the current backend needs them.
   - Persist `isMuted` on the voice session.
   - Add a narrow mute endpoint that emits safe voice events.
   - Keep mute as an overlay flag, not a replacement lifecycle state.

3. Refactor `/voice` UI toward Call Mode.
   - Mobile-first call entry with concise profile selection.
   - Large central status visual and short human-readable status.
   - Prominent mute control.
   - Max 4-5 primary visible actions during a call.
   - Text fallback and transcript moved behind secondary controls.
   - Handoff status shown after call end; if direct Telegram open is not wired, show a placeholder honestly.

4. Improve browser-level call behavior.
   - Prefer immediate short Realtime auto-greeting on call start.
   - Implement browser-level barge-in approximation: when user speech starts while Hermes is speaking, cancel the current response/audio where the current browser/OpenAI event surface allows it, return UI to listening, and document the remaining true full-duplex gap.
   - Make retrieval/research/tool statuses short and non-debug.

5. Update tests.
   - Cover V1 profile exposure and Telegram binding serialization.
   - Cover mute endpoint and no raw context leakage.
   - Cover Call Mode UI expectations: call-first language, prominent mute, secondary text/transcript/details.

6. Verify.
   - `npx tsx --test tests/voice/*.test.ts*`
   - `npm run build`

## Known V1 Gaps To Document In Code

- True realtime full-duplex barge-in depends on browser/OpenAI Realtime event behavior. V1 will stop/cancel the current response where possible and reflect listening immediately, but may not perfectly interrupt every provider-level audio frame.
- Full transcript/summary/memory/handoff extraction is currently backed by the existing `VOICE_CALL_MEMORY_V1` memory summary endpoint; richer Decisions/Produces/Tags extraction remains a follow-up.
- A direct Telegram deep-link button is shown only when a safe binding is available. Otherwise the UI shows handoff readiness without pretending a complete Telegram action is wired.
