🔴 Critical
Hardcodierte Fallback-Credentials in Production
🔴 Critical
middleware.ts · app/api/auth/login/route.ts
Beide Dateien nutzen ?? "mc-dev-secret-change-in-production" bzw. ?? "mission2024" als Fallback, wenn die Umgebungsvariablen MC_JWT_SECRET und MC_PASSWORD nicht gesetzt sind. Falls die Env-Vars fehlen oder leer sind — z.B. durch einen Deploy-Fehler in Coolify — kann jeder mit dem Passwort mission2024 einloggen und beliebige JWT-Tokens fälschen (da das Secret bekannt ist).
💡 Empfehlung

Hard-fail statt Fallback: if (!process.env.MC_JWT_SECRET) throw new Error("MC_JWT_SECRET is required"). App-Start schlägt fehl statt mit schwachen Credentials hochzukommen. Zusätzlich: starkes Secret (≥32 Byte random) und starkes Passwort in Coolify setzen, beide Vars als "required" dokumentieren. Passwort sofort ändern.

Kein Rate-Limiting auf dem Login-Endpoint
🔴 Critical
app/api/auth/login/route.ts
Der POST /api/auth/login Endpoint hat keinerlei Brute-Force-Schutz. Ein Angreifer kann unbegrenzt Passwort-Versuche senden — bei einem 6-stelligen numerischen Passwort wären das ~1 Mio. Requests, bei einem Wörterbuch-Angriff noch schneller. Keine Lockout-Logik, kein CAPTCHA, keine IP-Drosselung.
💡 Empfehlung

In-Memory-Rate-Limiter per IP implementieren (z.B. max 5 Versuche / 15 Minuten), danach exponentielles Backoff. Alternativ: Reverse-Proxy (Coolify/Nginx/Traefik) mit Rate-Limit-Middleware vorschalten. Kurzfristig: IP-Allowlist im Reverse-Proxy auf bekannte IPs beschränken.

🟠 High
Gesamter OpenClaw-Workspace via /api/memory exponiert
🟠 High
docker-compose.yml · app/api/memory/route.ts · lib/fs.ts
MEMORY_DIR=/workspace und BRIEFINGS_DIR=/workspace zeigen beide auf das gesamte ~/.openclaw/workspace-Verzeichnis. Die listMemory()-Funktion gibt alle .md-Dateien darin zurück — das sind u.a. SETUP.md (API Keys, Infrastruktur-Details), MEMORY.md (persönliche Daten), USER.md (private Infos), SOUL.md und alle Tagesnotizen. Jeder authentifizierte Nutzer (also jeder mit dem Login-Passwort) kann diese Dateien lesen. Bei Kombination mit C2 (Brute-Force): kein weiterer Auth-Aufwand nötig.
💡 Empfehlung

MEMORY_DIR und BRIEFINGS_DIR auf dedizierte Unterverzeichnisse einschränken, z.B. /workspace/briefings und /workspace/memory-exports (nur explizit freigegebene Dateien). Niemals die gesamte Workspace-Root exponieren. Volume im docker-compose entsprechend anpassen: /home/hartner/.openclaw/workspace/briefings:/workspace/briefings:ro.

API-Routes ohne Defense-in-Depth Auth
🟠 High
app/api/*/route.ts (alle Routes)
Alle API-Routes (/api/briefings, /api/memory, /api/tasks, /api/projects, /api/people, /api/hatti) verlassen sich ausschließlich auf die Middleware für Authentifizierung. Keiner der Handler verifiziert selbst das JWT. Wenn die Middleware einen Route-Matcher-Bug hat, ein Future-Next.js-Update das Verhalten ändert oder ein Route-Handler-Pfad nicht vom Matcher erfasst wird, sind alle Daten ungeschützt. Defense in Depth fehlt komplett.
💡 Empfehlung

Hilfs-Funktion verifyAuth(request) erstellen, die das JWT aus dem Cookie prüft und 401 zurückgibt wenn ungültig. In jeden Route-Handler als ersten Aufruf einbauen. Middleware bleibt als erste Verteidigungslinie, Route-Handler als zweite.

Unvalidiertes Object-Spread in POST/PATCH — Prototype Pollution / Field Injection
🟠 High
app/api/projects/route.ts · app/api/people/route.ts · app/api/tasks/route.ts
Alle drei Routes nutzen const project = { ...body, id: Date.now().toString() } bzw. projects[idx] = { ...projects[idx], ...body } ohne Schema-Validierung. Ein Angreifer kann beliebige Felder einfügen (z.B. __proto__, constructor) oder bestehende Objekte durch PATCH komplett überschreiben — inklusive der id (wird beim PATCH nicht überschrieben, aber andere Felder schon). Bei PATCH auf Tasks kann z.B. der Status auf einen ungültigen Wert gesetzt werden.
💡 Empfehlung

Zod oder manuelle Allowlist-Validierung implementieren: nur explizit erlaubte Felder aus dem Body extrahieren. Beispiel: const { title, project, notes } = await req.json() statt ...body. Für PATCH analog: nur bekannte Felder patch-en.

Keine HTTP-Security-Headers konfiguriert
🟠 High
next.config.mjs
In next.config.mjs sind keine Security-Headers gesetzt. Next.js liefert by default: kein Content-Security-Policy, kein X-Frame-Options, kein X-Content-Type-Options, kein Referrer-Policy, kein Permissions-Policy, kein Strict-Transport-Security. Das macht die App anfällig für Clickjacking, MIME-Sniffing und erleichtert XSS-Angriffe erheblich.
💡 Empfehlung

In next.config.mjs den headers()-Block ergänzen: X-Frame-Options: DENY, X-Content-Type-Options: nosniff, Referrer-Policy: strict-origin-when-cross-origin, Strict-Transport-Security: max-age=63072000; includeSubDomains, sowie eine restriktive CSP. Verifikation via securityheaders.com.

Next.js mit bekannten DoS-Vulnerabilities (CVE)
🟠 High
package.json · next@14.2.35
npm audit meldet 4 High-Severity-Vulnerabilities in der installierten Version. Konkret: GHSA-h25m-26qc-wcjf — Next.js HTTP-Request-Deserialisierung kann bei unsicheren React Server Components zu DoS führen. GHSA-9g9p-9gw9-jx7f — Image Optimizer DoS via remotePatterns. Diese betreffen direkt Next.js selbst und sind bei self-hosted Deployments relevant.
💡 Empfehlung

Next.js auf die neueste stabile 15.x Version updaten (npm install next@latest). Die Breaking Changes zwischen 14→15 sind überschaubar (Async Params in Route Handlers). Danach npm audit erneut prüfen. glob-Vulnerability in devDependencies ist geringes Risiko (build-time only).

🟡 Medium
iframe Preview von KI-generiertem HTML ohne Sandbox
🟡 Medium
app/docs/page.tsx:294 · app/api/briefings/route.ts
Briefings werden als rohes HTML direkt in einem <iframe src="/api/briefings?file=...&raw=1"> angezeigt — ohne sandbox-Attribut. Das HTML ist KI-generiert und hat Zugriff auf JavaScript, localStorage, kann fetch-Requests abfeuern und auf Cookies zugreifen. Da der iframe auf demselben Origin läuft, hat er vollen Zugriff auf die App-Session.
💡 Empfehlung

sandbox="allow-scripts allow-same-origin" → lieber sandbox="allow-scripts" (ohne same-origin!). Oder: Briefings auf einem anderen Origin / subdomain ausliefern (briefings.mehlhart.de). Zusätzlich: CSP-Header auf dem raw=1-Endpoint setzen, der script-src einschränkt.

30-Tage JWT ohne Revocation-Mechanismus
🟡 Medium
app/api/auth/login/route.ts
Tokens sind 30 Tage gültig. Es gibt kein Logout, keinen Revoke-Endpoint, keine Token-Rotation, keine Session-Blacklist. Ein gestohlenes Cookie bleibt bis zu 30 Tage nutzbar. Es gibt keine Möglichkeit, nach Kompromittierung alle aktiven Sessions zu invalidieren ohne das JWT-Secret zu ändern (was alle anderen Sessions ebenfalls invalidiert).
💡 Empfehlung

Token-Laufzeit auf 24h reduzieren + Refresh-Token-Mechanismus implementieren. Alternativ: Logout-Endpoint hinzufügen, der ein "issued-before"-Timestamp in den JWT-Claims prüft (server-side Revocation ohne DB). Kurzfristig: 7 Tage statt 30 Tage als Kompromiss.

Interne Error-Messages werden an den Client geleakt
🟡 Medium
app/api/briefings/route.ts · app/api/memory/route.ts
Im Catch-Block wird (error as Error).message direkt an den Client zurückgegeben. Das kann interne Dateipfade, Stack-Traces oder Systeminfos enthalten — wertvolle Informationen für einen Angreifer (z.B. "ENOENT: no such file or directory, open '/home/hartner/.openclaw/workspace/...").
💡 Empfehlung

Im Catch-Block nur generische Meldungen zurückgeben: { error: "Failed to read file" }. Intern loggen (server-side) mit vollem Error. Niemals rohe error.message oder Stack-Traces an den Client senden.

Fallback-Datenspeicher in /tmp (world-readable)
🟡 Medium
app/api/tasks/route.ts · app/api/projects/route.ts · app/api/people/route.ts
Wenn /data/briefings/ nicht beschreibbar ist, fallen alle drei Routes auf /tmp/mc-tasks.json, /tmp/mc-projects.json, /tmp/mc-people.json zurück. Das /tmp-Verzeichnis ist unter Linux typischerweise world-readable (0755) und wird nicht persistiert. Enthält Kontaktdaten (E-Mail-Adressen von Kunden), Projekt-Repos, Task-Details.
💡 Empfehlung

TASKS_FILE, PROJECTS_FILE, PEOPLE_FILE in Coolify/docker-compose korrekt setzen und auf ein Volume mounten. Fallback entweder ganz entfernen oder zumindest auf ein gesichertes Verzeichnis (z.B. /app/data/ im Container) umleiten. Beim Start prüfen ob Primary-Path schreibbar ist, sonst hard fail.

Dockerfile Build-Stage: npm ci --only=production vor npm run build
🟡 Medium
Dockerfile
Das Dockerfile führt npm ci --only=production aus, dann npm run build. TypeScript-Compilierung und ESLint (benötigt für den Build) sind devDependencies und werden dabei übersprungen. In der Praxis läuft der Build nur, weil node_modules/ vom Host-Checkout vorhanden ist. In einer sauberen CI/CD-Pipeline würde dieser Build fehlschlagen. Das ist ein Konfigurationsfehler der die Reproduzierbarkeit des Builds gefährdet.
💡 Empfehlung

Build-Stage: npm ci (ohne --only=production) — alle Dependencies für den Build. Runner-Stage: nur das fertige .next/standalone-Artefakt kopieren (wie jetzt). Alternativ zweistufig: npm ci → npm run build → npm prune --production.

🟢 Low / Info
SameSite=Lax statt Strict für Auth-Cookie
🟢 Low
app/api/auth/login/route.ts
Das Auth-Cookie ist auf sameSite: "lax" gesetzt. Für eine single-user App ohne Cross-Site-Navigation-Anforderungen wäre strict die sicherere Wahl — verhindert CSRF auch bei Top-Level-Navigation.
💡 Empfehlung

sameSite: "strict" setzen, sofern keine Redirect-Flows von externen Seiten gebraucht werden. Bei Mission Control (direkter Login) kein Problem.

Kein Audit-Log für Auth-Events
🟢 Low
app/api/auth/login/route.ts
Fehlgeschlagene und erfolgreiche Login-Versuche werden nicht geloggt. Bei einem Angriff gibt es keine Möglichkeit nachzuvollziehen, ob und wann die App kompromittiert wurde.
💡 Empfehlung

console.warn / console.error mit Timestamp und IP bei fehlgeschlagenen Logins. Docker-Logs werden von Coolify erfasst. Alternativ: strukturiertes Logging mit IP-Hashing (privacy-compliant).

Keine Input-Längen-Limits in API-Routes
🟢 Low
app/api/tasks/route.ts · app/api/hatti/route.ts
Task-Titel, Notes, Hatti-Messages haben keine maximale Länge. Ein Angreifer könnte Megabytes an Daten in die JSON-Dateien schreiben, was zu Disk-Full-Zustand oder Memory-Problemen führen kann.
💡 Empfehlung

Maximale String-Längen definieren: Titel max 200 Zeichen, Notes max 2000, Message max 5000. Validierung direkt nach dem JSON-Parse.

Port 3000 direkt in docker-compose exponiert
🟢 Low
docker-compose.yml
ports: "3000:3000" bindet den Container-Port an alle Interfaces des Hosts (0.0.0.0:3000). Wenn der Host eine öffentliche IP hat und kein Firewall-Rule greift, ist die App direkt (ohne TLS, ohne Coolify-Reverse-Proxy) erreichbar.
💡 Empfehlung

Port nur auf Localhost binden: "127.0.0.1:3000:3000". Der Reverse-Proxy (Coolify/Traefik) verbindet sich intern — kein externer Zugriff auf Port 3000 nötig. Ergänzend: Firewall-Regel auf dem Host (ufw allow/deny 3000).

✅ Das Gute — gut gemacht
✅ Non-Root Docker User — Container läuft als nextjs (UID 1001), nicht als root. Privilege Escalation erheblich erschwert.
✅ Volume Read-Only gemountet — /workspace:ro in docker-compose verhindert, dass die App in den OpenClaw-Workspace schreiben kann.
✅ httpOnly Cookie — Das Auth-Cookie ist httpOnly: true, damit kein JavaScript im Browser es lesen kann. Richtig gemacht.
✅ Secure Cookie in Production — secure: process.env.NODE_ENV === "production" — Cookie wird nur über HTTPS übertragen in prod.
✅ Path-Traversal-Schutz in lib/fs.ts — safeResolve() mit isWithinRoot()-Check verhindert ../../../etc/passwd-Angriffe. Gut implementiert.
✅ MAX_DEPTH = 2 in walkDir() — Verhindert tiefes Traversieren des Dateisystems bei Directory-Listing.
✅ jose statt jsonwebtoken — Moderne, Web-Crypto-basierte JWT-Bibliothek ohne bekannte Vulnerabilities. Algorithmus explizit auf HS256 gesetzt.
✅ Middleware-Matcher deckt alle Routes ab — /((?!_next/static|_next/image|favicon.ico).*) erfasst korrekt alle API-Routes und Pages.
✅ TypeScript durchgehend — Statische Typisierung reduziert eine ganze Klasse von Bugs. Kein JavaScript-Schludern.
✅ Docker Healthcheck vorhanden — Container wird automatisch neugestartet wenn der Health-Endpoint nicht antwortet.

Hardcodierte Credentials entfernen + starke Env-Vars erzwingen
Sofort: MC_PASSWORD und MC_JWT_SECRET in Coolify auf starke Werte setzen. Code auf Hard-Fail bei fehlenden Env-Vars umstellen. Passwort auf 16+ Zeichen erhöhen.

Rate-Limiting auf /api/auth/login implementieren
Max 5 Versuche / 15 Minuten per IP. In-Memory mit einer einfachen Map realisierbar, alternativ Traefik-Middleware in Coolify konfigurieren. Stops Brute-Force komplett.
🔴 Critical Impact 🔵 ~2h Aufwand
3
MEMORY_DIR / BRIEFINGS_DIR auf dedizierte Unterverzeichnisse beschränken
Workspace-Root aus dem Docker-Volume entfernen. Nur /workspace/briefings und ein explizites /workspace/memory-public-Verzeichnis mounten. Verhindert unbeabsichtigten Zugriff auf API Keys und persönliche Daten.
🟠 High Impact 💚 ~1h Aufwand
4
HTTP-Security-Headers in next.config.mjs hinzufügen
X-Frame-Options, X-Content-Type-Options, CSP, HSTS, Referrer-Policy. Standardisierter headers()-Block, fertige Templates verfügbar. Verbessert Security-Score von F auf A. Blockiert Clickjacking und MIME-Sniffing sofort.
🟠 High Impact 💚 ~45min Aufwand
5
Next.js auf 15.x updaten + iframe-Sandbox hinzufügen
Next.js-Update schließt zwei aktive DoS-Vulnerabilities. iframe-Sandbox (sandbox="allow-scripts" ohne same-origin) isoliert KI-generierten HTML-Content und verhindert Session-Hijacking via Briefing-XSS. Beides im gleichen Zug machbar.
🟠 High Impact 🔵 ~3h Aufwand
