Stand: 10. März 2026 · Tenant-basiertes Seat-Billing

User-Flow

Registrierung → Login → Abo-Check │ ┌─────────┴─────────┐ │ │ Abo aktiv Kein Abo │ │ ▼ ▼ Dashboard /subscribe Seite │ ▼ Stripe Checkout │ ▼ Zahlung OK → Webhook │ ▼ DB: status = active │ ▼ Dashboard ✓ Monatlich: Stripe zieht automatisch ein. Bei Fehlschlag/Kündigung → Webhook → status = inactive → Paywall.

Architektur

KomponenteFunktion
subscriptions Tabelletenant_id, stripe_customer_id, stripe_subscription_id, status, current_seats
create-checkout Edge FnZählt Seats im Tenant → Stripe Checkout Session mit quantity = seat_count
stripe-webhook Edge FnEmpfängt Stripe Events → aktualisiert subscriptions.status in DB
Paywall-Guard (AppLayout)Nach Login: status ≠ 'active' → Redirect zu /subscribe
Seat-SyncBei invite / delete team-member → Stripe quantity updaten
Billing ist Tenant-basiert: Admin zahlt, alle User im Büro haben Zugang.
Seat-Count = COUNT(*) FROM profiles WHERE tenant_id = X

Preismodelle

ModellBeispielVorteilNachteil
A: Per-Seat Flat 80 €/Seat/Monat Transparent, linear Kein Mengenrabatt
B: Graduated (Staffel) 1–5: 89 €, 6–20: 69 €, 21+: 49 € Anreiz für Wachstum Komplexer zu kommunizieren
C: Flat-Tier 1–5 = 400 €, 6–20 = 1.200 € Einfach, planbar Große Preissprünge
Bei A und B: Stripe berechnet automatisch anhand der quantity.
Bei C: App muss Tier manuell wechseln.
