Skills & Plugins

Wie du Claude für KHB Ingenieure maßschneidern kannst — konkret und ohne Programmierkenntnisse.

📅 27. März 2026 ✍️ Von Daniel an Kais ⏱ ~8 Min. Lesen

Hey Kais, du hast jetzt ein Bild davon, was Claude Cowork grundsätzlich kann. Jetzt gehen wir einen Schritt tiefer: das Plugin-System ist der entscheidende Unterschied zwischen "einem generischen KI-Assistenten" und "deinem KHB-Assistenten, der weiß wie eure Projekte aufgebaut sind".

Ich zeig dir wie das technisch funktioniert — und wie ihr in einem Nachmittag ein Plugin bauen könnt, das sich anfühlt als ob Claude seit Jahren bei KHB arbeitet.

01 / Grundlagen

Was sind Plugins in Claude Cowork?

Stell dir vor, du hast gerade Claude Desktop installiert — er ist klug, aber er kennt KHB noch nicht. Er weiß nichts von eurer Ordnerstruktur, den HOAI-Leistungsphasen, euren Abläufen. Er ist wie ein sehr fähiger Neuer am ersten Arbeitstag.

Ein Plugin ändert das. Du installierst "KHB Projektmanagement" und plötzlich kennt Claude eure V3-Ordnerstruktur, weiß was LPH 3 bedeutet, und kann mit einem Befehl den Projektstatus für alle 37 aktiven Projekte abrufen. Wie eine App auf dem Smartphone — du installierst "Kamera" und dein Telefon kann plötzlich fotografieren.

02 / Aufbau

Die 4 Bausteine eines Plugins

Jedes Plugin besteht aus denselben vier Komponenten. Du kannst alle vier nutzen — oder nur die, die du brauchst.

📋

Skills

skills/ Ordner

Markdown-Dateien die Wissen und Workflows enthalten. Claude liest sie beim Start und verhält sich entsprechend. Dein Firmen-Know-how als Text.

„Unsere Projektordner sind nach dem V3-Schema aufgebaut: Eingang / Ausgang / Bearbeitung..."
⚡

Slash Commands

commands/ Ordner

Markdown-Dateien die Schnellbefehle definieren. Du tippst /projektstatus und Claude führt einen definierten Workflow aus — ohne langes Erklären.

/briefing → Claude erstellt Top-5-Aufgaben für heute aus allen aktiven Projekten
🔌

Connectors

.mcp.json

Verbinden Claude mit externen Tools. MCP (Model Context Protocol) ist ein offener Standard. Für KHB relevant: SharePoint, Outlook, Excel, Lexoffice.

SharePoint-Connector → Claude kann direkt in euren Projektordnern lesen und suchen
🤖

Sub-Agents

agents/ Ordner

Spezialisierte Mini-Agenten für delegierte Aufgaben. Claude teilt komplexe Jobs auf und gibt Teilaufgaben weiter — Recherche, Dateiorganisation, Analyse.

„Recherchiere für Projekt X alle Normen und fasse sie zusammen" → delegiert an Sub-Agent
03 / Architektur

So sieht ein Plugin aus — von innen

Hier die komplette Ordnerstruktur. Alles Markdown — kein Code, keine Programmierung. Wer Word-Dokumente schreiben kann, kann auch ein Plugin bauen.

mein-plugin/
├── plugin.json         ← Name, Version, Beschreibung des Plugins
├── .mcp.json           ← Connector-Konfigurationen (SharePoint, Outlook, ...)
│
├── commands/
│   ├── projektstatus.md  ← Wird zu /projektstatus im Chat
│   ├── briefing.md       ← Wird zu /briefing im Chat
│   └── abgabe-check.md   ← Wird zu /abgabe-check im Chat
│
└── skills/
    ├── projektstruktur.md ← V3-Ordnerstruktur, erklärt für Claude
    ├── lph-phasen.md      ← HOAI LPH 1–9, Deliverables pro Phase
    └── abgabe-checkliste.md ← Was vor Abgaben geprüft wird

💡 Die plugin.json ist ein paar Zeilen JSON (Name, Beschreibung, Version) — Claude kann sie für dich generieren. Die Skills und Commands schreibst du auf Deutsch, in normaler Sprache.

05 / Setup

Eigenes KHB-Plugin bauen — so geht's

Du brauchst keinen Entwickler dafür. Der Prozess ist bewusst conversational gehalten.

  • 1

    Plugin Management installieren

    Claude Desktop öffnen → Cowork Tab → Plugins → "Plugin Create" installieren. Das ist ein eingebautes Meta-Plugin das dir beim Bauen hilft.

  • 2

    Claude briefen — ganz normal im Chat

    Schreib: "Erstelle mir ein Plugin für ein Ingenieurbüro mit 10 Personen. Wir brauchen Skills für Projektstatusabfragen, Morgen-Briefings, und Dokumentensuche in SharePoint."

  • 3

    Claude führt dich durch den Setup

    Conversational, kein Code nötig. Claude stellt Fragen, generiert die Dateien, du überprüfst und passt an. Dauert ca. 30–60 Minuten.

  • 4

    Alternativ: Starter-Plugin nehmen und anpassen

    Operations-Plugin von GitHub laden, die skills/ Dateien durch KHB-spezifische ersetzen. Schneller Weg wenn du weißt was du willst.

  • 5

    Skills in Markdown schreiben

    Einfach auf Deutsch beschreiben: "Unsere Projektstruktur ist V3, Ordner heißen Eingang/Ausgang/Bearbeitung..." — Claude kann das auch für dich schreiben wenn du es ihm erklärst.

  • 6

    Slash Commands testen

    Plugin aktivieren, /briefing eintippen, schauen ob Claude das tut was du erwartest. Iterieren bis es passt.

  • 7

    Optional: Im Team verteilen

    Mit Enterprise Plan kannst du das Plugin für alle 10 Personen im Büro bereitstellen — via internem Marketplace oder direktes Teilen des Plugin-Ordners.

06 / Konkretes Beispiel

So könnte das KHB-Plugin aussehen

Hier ein konkreter Entwurf was "KHB Projektmanagement" als Plugin enthalten würde — das wäre mein Vorschlag als Startpunkt:

🏗️

KHB Projektmanagement

Custom Plugin · Version 1.0 · Für KHB Ingenieure GmbH

📋 Skills (Firmenwissen)

  • projektstruktur-v3.md → Beschreibt die V3-Ordnerstruktur: Eingang / Ausgang / Bearbeitung — Claude weiß wo welche Dateien liegen
  • lph-workflow.md → HOAI-Leistungsphasen 1–9 mit typischen Deliverables pro Phase. Claude versteht was "wir sind in LPH 4" bedeutet
  • abgabe-checkliste.md → Was vor einer Abgabe geprüft werden muss. Welche Dokumente, welche Freigaben, welche Formate
  • khb-style-guide.md → Wie KHB Dokumente formatiert — Namenskonventionen, Kopfzeilen, Revisionsstände

⚡ Slash Commands (Schnellbefehle)

  • /projektstatus [Name] → Scannt Projektordner, zeigt offene Punkte, nächste Deadline, letzten Fortschritt
  • /briefing → Morgen-Briefing mit Top-5 Aufgaben aus allen aktiven Projekten — täglich als Start
  • /abgabe-check [Name] → Prüft Vollständigkeit vor Abgabe: sind alle Pläne vorhanden, Checkliste abhaken
  • /suche [Begriff] → Durchsucht Projektordner und SharePoint nach Vorlagen, alten Dokumenten, Beispielen

🔌 Connectors (Externe Tools)

📁 SharePoint (M365) — Projektordner lesen 📧 Outlook (M365) — Projekt-relevante Mails 📊 Excel — Stammdaten der 37 Projekte
07 / Ehrlichkeit

Was du wissen solltest — die Grenzen

Ich will transparent sein, was das System noch nicht kann oder wo es Aufwand gibt:

💳

Bezahlter Plan nötig

Custom Plugins laufen nur auf Pro ($20/Mo) oder höher. Der Free Plan hat keinen Plugin-Support.

💾

Lokal gespeichert

Plugins liegen lokal auf dem Gerät — kein automatischer Cloud-Sync zwischen Geräten. Muss man manuell synchronisieren.

🎯

Klare Prompts nötig

Sub-Agents brauchen präzise Anweisungen. Bei vagen Beschreibungen improvisiert Claude — manchmal falsch.

🏢

Team-Features brauchen Enterprise

Interner Marketplace, Auto-Install, zentrale Verwaltung für alle 10 Personen — das ist Enterprise-Terrain ($$$).

🔄

Skills müssen gepflegt werden

Wenn sich Prozesse ändern, muss die Markdown-Datei angepasst werden. Kein großer Aufwand, aber jemand muss es tun.

🔐

Connector-Setup braucht IT

Den M365-Connector für SharePoint einrichten braucht einmalig jemanden mit IT-Zugang. Danach läuft es.

Mein Vorschlag für den Einstieg

Du kannst das binnen eines Nachmittags haben. Hier der konkrete Plan:

1
Operations-Plugin von GitHub als Basis nehmen — 5 Minuten
2
2–3 Skills auf KHB anpassen: Projektstruktur, LPH, Briefing — 1–2 Stunden
3
/projektstatus und /briefing einrichten und testen — 30 Minuten
4
Erst auf deinem Rechner testen, dann für das Team freigeben
