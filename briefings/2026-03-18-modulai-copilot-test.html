🛠️ Code Review — Potential Issues aus PR-Analyse
🔴
Feature-Flag fehlt im Frontend (PR #30)
Die Edge Function hat eine Feature-Flag-Prüfung, aber das Frontend zeigt den Copilot-Button ohne Flag-Check in ProjectDetailPage. Falls der Flag aus ist, erscheint der Button trotzdem. → Flag-Check auch im Button-Render einbauen.
🔴
Route /copilot vs. /projekte — Inkonsistenz
Die bestehende App nutzt deutsche Routen (/projekte, /aufgaben). PR #31 fügt /copilot (englisch) hinzu. Das sollte konsistent sein — entweder alles englisch oder alles deutsch. Empfehlung: /copilot → /kopilot oder alle Routen auf Englisch.
🟡
Context-Window Default: 7 Tage vs. PR-Beschreibung "4 Wochen max"
PR #31 beschreibt einen Selektor mit 1/2/4 Wochen und Default 7 Tage. Die Edge Function aus PR #30 fetcht hingegen 90 Tage Tasks und 30 Tage Time Entries (hartkodiert). Diese Werte sollten aus dem context_days Parameter kommen — prüfen ob der neue Wert korrekt durchgereicht wird.
🟡
Conversation-Titel: Kein Auto-Title in PR #31
Die DB hat ein title-Feld für Conversations, aber im PR-Code ist kein Auto-Titel aus der ersten Nachricht beschrieben. Conversations heißen vermutlich alle "Neues Gespräch" → schlechte UX bei mehreren Chats. Empfehlung: Ersten 40 Zeichen der ersten User-Nachricht als Titel.
🟡
Mobile: Conversation-Sidebar auf kleinen Screens
PR #31 beschreibt ein 2-Spalten-Layout (Sidebar + Chat). Auf Mobile (390px) braucht das einen Toggle-Mechanismus. Aus dem PR-Code geht nicht hervor ob das ein Sheet/Drawer wird oder ob die Sidebar einfach versteckt ist. → Prüfen nach Deploy.
🟢
Error Handling: Edge Function hat gutes Error-Handling
Die Edge Function prüft Auth, Project-Access und gibt sinnvolle HTTP-Status-Codes zurück. Frontend-seitig sollte das in der Hook aufgegriffen werden — im PR-Code vorhanden (useProjectCopilot hat error-State). Gut umgesetzt.
🟢
RLS Policies: Korrekt umgesetzt
Die DB-Migration enthält Row-Level-Security für copilot_conversations und copilot_messages. User sieht nur eigene Conversations. Sauber.
