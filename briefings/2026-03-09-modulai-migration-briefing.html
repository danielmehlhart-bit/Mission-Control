ModulAI: Lovable → Hetzner/Coolify

Frontend-Migration — selbst gehosted, volle Kontrolle, Supabase bleibt komplett unangetastet.

🚫 Lovable Frontend (wird abgelöst) ✨ Coolify auf Hetzner (neu) ✓ Supabase (bleibt komplett) 🔗 github.com/danielmehlhart-bit/modulai

🎯 1. Überblick & Ziel

ModulAI läuft aktuell auf Lovable — einem Service der zwei Dinge gleichzeitig macht: Frontend hosten + KI-gestützte UI-Generierung. Das Backend (Auth, Datenbank, Edge Functions, Storage) liegt schon jetzt bei Supabase und ist vollständig unabhängig von Lovable.

Das Ziel: Das Frontend von Lovable lösen und auf einem eigenen Hetzner-Server via Coolify betreiben. Coolify übernimmt Deployments, SSL, und Domain-Routing — vollautomatisch, ohne DevOps-Kenntnisse.

🔍 2. Was Lovable macht — und was nicht migriert werden muss

Lovable ist im Kern zwei Dinge: eine KI-IDE (Code generieren) und ein statischer Hoster (dein Build online stellen). Mehr nicht.

❌ Was Lovable NICHT ist:

  • Kein Backend — Lovable kennt deine Supabase-Daten nicht
  • Kein Edge-Function-Runner — die laufen in Supabase's eigenem Netz
  • Kein Datenbankserver — Postgres ist komplett bei Supabase
  • Kein Auth-Provider — das macht Supabase

✅ Was beim Umzug relevant ist:

  • Nur das Frontend-Bundle (HTML + JS + CSS nach npm run build)
  • Die Env-Variablen (VITE_SUPABASE_URL und VITE_SUPABASE_ANON_KEY)
  • Die Domain-Konfiguration in Supabase (Redirect URLs)
🎉 Die gute Nachricht

Du migrierst nur den Hoster, nicht das Backend. Wenn Coolify läuft und die Env-Variablen stimmen, funktioniert alles wie vorher — Auth, DB, Functions, alles. Die meiste Arbeit ist Setup, nicht Migration.

✅ 3. Voraussetzungen checken

Bevor du startest, diese 4 Dinge prüfen:

  • GitHub Repo ist aktuell — Lovable → GitHub Sync aktiviert, letzter Commit ist dein aktueller Stand. URL: github.com/danielmehlhart-bit/modulai
  • Hetzner Server läuft — Coolify ist installiert und erreichbar (normalerweise über Port 8000 oder deine Server-IP). Falls noch nicht: Coolify-Installer (curl -fsSL https://cdn.coollabs.io/coolify/install.sh | bash)
  • Domain/Subdomain bereit — z.B. modulai.mehlhart.de. Du brauchst Zugang zu deinen DNS-Einstellungen (z.B. bei Cloudflare, IONOS, Strato).
  • Supabase Keys bereit — Supabase Dashboard → Projekt → Settings → API. Du brauchst: Project URL und anon public key.

🚀 4. Schritt-für-Schritt Migration

A
GitHub Repo vorbereiten
📁 Repo-Check

Prüfe ob dein Repo die richtigen Build-Dateien enthält. Bei einem Lovable-Projekt ist das fast immer ein Vite-Setup.

  • Im Repo prüfen: package.json und vite.config.ts vorhanden?
  • In package.json nach dem scripts-Block schauen — Build-Befehl ist dort
  • Typisch: npm run build → Output in dist/
  • .env.example anlegen (zum Dokumentieren, nicht für echte Keys!)
# .env.example
VITE_SUPABASE_URL=https://dein-projekt.supabase.co
VITE_SUPABASE_ANON_KEY=dein-anon-key-hier
# package.json scripts (Beispiel)
{
  "scripts": {
    "build": "vite build",
    "dev": "vite"
  }
}
B
Coolify Setup auf Hetzner
✨ Neu auf Coolify
  • Coolify öffnen → New Project erstellen (z.B. "ModulAI")
  • Im Projekt: + New Resource klicken
  • Typ wählen: "Static Site" (wenn Lovable ein Vite/React-SPA ist — das ist der Normalfall)
  • GitHub Source wählen → Repo danielmehlhart-bit/modulai verbinden (OAuth empfohlen)
  • Branch: main
  • Build-Command eintragen: npm run build
  • Publish Directory (Output): dist
# Coolify Build-Einstellungen
Build Command:    npm run build
Publish Directory: dist
Node Version:     20 (oder latest LTS)
C
Environment Variables in Coolify setzen
⚠️ Wichtig!

Das ist der häufigste Fehler: Env-Variablen vergessen. Vite baut sie zur Build-Zeit in den Bundle ein — sie müssen vor dem Deploy gesetzt sein.

  • In Coolify: Resource → Environment Variables Tab
  • Beide Variablen aus Supabase Dashboard → Settings → API kopieren
  • Achtung: Variablen MÜSSEN mit VITE_ beginnen, sonst ignoriert Vite sie!
VITE_SUPABASE_URL=https://abcdefghijk.supabase.co
VITE_SUPABASE_ANON_KEY=eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...
D
Supabase CORS & Allowed Origins updaten
✓ Supabase Aktion
  • Supabase Dashboard → dein Projekt → Authentication → URL Configuration
  • Site URL auf neue Domain setzen: https://modulai.mehlhart.de
  • Unter Redirect URLs: neue URL hinzufügen: https://modulai.mehlhart.de/**
  • Alten Lovable-URL als Redirect URL kann vorerst bleiben (schadet nicht)
# Supabase Authentication → URL Configuration
Site URL:      https://modulai.mehlhart.de
Redirect URLs: https://modulai.mehlhart.de/**
               https://modulai.mehlhart.de/auth/callback
E
Custom Domain in Coolify konfigurieren
🌎 DNS
  • In Coolify: Resource → Domains Tab
  • Domain eintragen: modulai.mehlhart.de
  • In deinem DNS-Anbieter: A-Record anlegen
  • SSL: Coolify macht Let's Encrypt automatisch — kein manuelles Zertifikat nötig!
# DNS-Einstellung bei deinem Provider
Typ:   A
Name:  modulai
Wert:  DEINE-HETZNER-SERVER-IP
TTL:   300 (5 Minuten für schnelles Testen)
F
Deploy & Test
✅ Go Live
  • In Coolify: "Deploy" Button klicken → Build-Log beobachten
  • Nach erfolgreichem Build: https://modulai.mehlhart.de im Browser öffnen
  • Login-Flow testen — geht Auth? Wird nach Login korrekt redirected?
  • Edge Functions testen (sollten sofort klappen — Supabase-seitig unverändert)
  • Alle Kernfunktionen einmal durchklicken
  • Browser-Konsole auf Fehler prüfen (F12 → Console)
G
Lovable deaktivieren / Domain ummelden
🚫 Lovable abschalten
  • In Lovable: Einstellungen → Custom Domain → Domain entfernen
  • DNS-Eintrag für Lovable-URL entfernen (falls vorhanden)
  • Supabase Redirect URL für Lovable-Domain kann entfernt werden
  • Optional: Lovable Projekt in "Published" lassen oder archivieren (Backup)
⚠️ Reihenfolge wichtig: Erst sicherstellen dass Coolify-Deploy läuft, DANN Lovable-Domain entfernen. Nicht umgekehrt.

⚡ 5. Edge Functions — fast nichts zu tun

✓ Kein Handlungsbedarf für bestehende Functions

Supabase Edge Functions laufen in Supabase's eigenem Deno-Runtime — vollständig unabhängig von Lovable oder Coolify. Die laufen weiter wie bisher.

  • Functions werden über die Supabase-URL aufgerufen — die ändert sich nicht
  • Aufrufe kommen aus dem Frontend via VITE_SUPABASE_URL — wird korrekt geladen
  • Kein Deployment, kein Refactoring, kein Konfigurationsaufwand

Falls Functions noch nicht deployed sind:

# Supabase CLI (einmalig installieren falls noch nicht vorhanden)
npm install -g supabase

# Login
supabase login

# Function deployen
supabase functions deploy name-der-function

# Alle Functions deployen
supabase functions deploy

⚠️ 6. Häufige Fallstricke

🚫 Env-Variablen fehlen oder falsch gesetzt

Die App startet, aber Auth schlägt fehl oder Daten werden nicht geladen. Fix: In Coolify → Environment Variables prüfen. Vite baut Env-Variablen zur Build-Zeit ein — nach Änderung immer neu deployen!

🚫 CORS / Redirect URLs in Supabase nicht aktualisiert

Login schlägt fehl, Browser zeigt "invalid redirect URL" oder Auth-Callback funktioniert nicht. Fix: Supabase → Authentication → URL Configuration → Site URL und Redirect URLs auf neue Domain setzen.

🚫 Falsches Build-Output-Verzeichnis in Coolify

Coolify sucht in build/ oder out/, aber Vite produziert dist/. Fix: Coolify → Resource → Build-Einstellungen → Publish Directory auf dist setzen.

🚫 Lovable-Domain noch aktiv — Duplicate Content

Wenn deine App gleichzeitig unter der Lovable-URL und deiner neuen Domain erreichbar ist, entstehen SEO-Probleme (Google sieht doppelte Inhalte). Fix: Lovable-Custom-Domain entfernen sobald Coolify-Deploy stabil läuft.

🚫 DNS-Propagation braucht Zeit

Neuer A-Record ist gesetzt, aber die Domain löst noch auf die alte IP auf. Fix: Warten (5–60 Min bei TTL 300). Prüfen mit: nslookup modulai.mehlhart.de

💡 Quick-Win Strategie

Fang mit einem Test-Deploy auf einer Subdomain an (z.B. modulai-staging.mehlhart.de) während Lovable noch läuft. So kannst du in Ruhe testen, ohne die laufende App zu gefährden. Erst wenn alles grün ist → Lovable abschalten.
